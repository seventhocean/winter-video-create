# Winter Video Create

## 目标

把“人脸口播 + 口播稿 + 素材文件夹 + 多层动画信息舞台”做成可复用的视频生产流程。每一段口播节奏都是一个舞台：舞台包含主视觉、辅助信息、真实素材和环境动效；先保证音画同步、信息准确、人物构图和文字清晰，再增加动效丰富度。

## 风格系统

把“内容语义、舞台空间、视觉风格、渲染引擎”分成四层管理。风格只改变颜色、字体、材质、运动曲线和密度，不改变口播时间轴、舞台语义、素材优先级或操作保护区。

- 风格规范见 [style-profiles.md](references/style-profiles.md)。
- 当前唯一已归档风格为 `clinical-tech`；不存在的风格不预设名称、不占位、不列入候选列表。
- 每条视频只使用一个已归档 `styleProfile`；章节只能切换强调色和环境色。明显换风格必须作为一次舞台转场记录。
- `styleProfile` 必须写入 `work/制作规格.md` 或 `timeline.json`，并由组件读取 tokens；禁止在场景代码里散落硬编码颜色和字体。
- 用户要求新的视觉方向时，先把它作为当前项目的实验性制作规格，不立即命名为全局风格。完成真实成片、至少一次复用，并提炼出稳定 tokens、组件适配和验收结果后，才归档为新的 Profile。

## 默认决策

- **声音是时间轴**：以真实口播的停顿、换句和关键词落点安排画面，不按总时长平均切卡片。
- **一句口播绑定一个主动作**：说到“字体”展示字体，说到“间距”调整间距；同一舞台可以同时承载 1 个主视觉、2–4 个辅助元素和 1 个低注意力环境动效，但不能堆无语义装饰。
- **每次转场先切换舞台，再编排元素**：先决定这一段是数据舞台、流程舞台、对比舞台、素材舞台、实操舞台还是全屏 B-roll，再决定每个元素何时进入、停留、强调和退出。
- **先读取已存在风格，再做舞台视觉化**：在分镜阶段记录已归档 `styleProfile`，先加载风格 tokens 和运动语法，再设计卡片、路径、字体和环境层；新的视觉尝试只记录在当前项目，不提前登记为风格。
- **舞台必须形成空间关系**：普通解释舞台至少使用 3 个空间锚点，覆盖至少 2 个横向区域和 2 个纵向区域；用路径、阶梯、环形进度、堆叠或状态线连接元素。禁止把主视觉、辅助信息和状态全部塞进同一个右上角 / 左上角容器。
- **禁止角落卡片退化**：如果连续两个舞台都表现为“人物不动 + 同一角落弹出一张玻璃卡”，立即停止实现并重写布局。卡片可以是一个图层，不能代替整个舞台。
- **多元素不等于同时抢注意力**：多个元素可以同时存在，但独立的新信息不要在同一瞬间首次出现；主元素先落位，辅助元素错开约 0.08–0.30 秒或保持低对比度。
- **素材文件夹是第一等输入**：把口播视频、口播稿和对应图片 / 视频素材放在同一项目文件夹；先盘点并建立素材语义索引，不凭感觉生成替代画面。
- **人物是叙事主体**：讲观点时保留人物；右侧需要信息时让人物平滑向左；每次都检查人物脸部、信息层和字幕安全区是否重叠。
- **全屏 B-roll 与信息层分开判断**：抽象观点、比喻和转折可以切入独立的全屏拼贴 B-roll；需要浮在人物画面上方的透明图形、标签或箭头，仍使用 Remotion / HyperFrames 信息层，不把拼贴 B-roll 当作透明动画层。
- **用户明确要求纯动画时，不使用截图作为解释性卡片**：用 Remotion 或 HyperFrames 绘制文章示意、流程、滑杆、浏览器框架、对比状态和高亮线。真实产品操作放到用户提供的录屏中。
- **文字必须正常方向且清晰**：不使用 `rotateY(180deg)`、镜像、3D 翻面或会导致文字倒置的父级变换。优先使用平移、缩放、淡入、线条生长、滑杆移动和 2D 页面切换。
- **画幅跟素材和平台走**：方形口播可以使用 `1080×1080`，抖音竖屏默认 `1080×1920`，不强制所有视频改成 `1920×1080`。
- **信息层不常驻同一个表格**：根据语义切换问题文字、文章示意、手动调整、发布对比、输入/预览流程等状态。
- **避免单卡片退化**：如果一个舞台只有一张孤立卡片，必须检查是否还能加入有语义的标题、标签、连线、数据状态、真实素材或章节进度；只有一句短观点、片尾和操作保护段可以例外。
- **实操画面的遮挡优先级固定**：原片已经包含真实操作模块时，按“操作模块 > 信息动效 > 人脸”处理。允许上半区动效遮挡人脸，禁止遮挡下半区的按钮、输入框、下拉菜单、光标和操作结果。
- **先区分快速预览与正式交付**：预览只验证 10–15 秒、首中尾关键帧和核心遮挡关系；风格确认后再做整片高质量检查，不在预览阶段执行完整成片 QA。

### 整屏舞台最低标准

除片尾、短句、实操保护段和全屏素材外，每个舞台同时满足：

1. 只有 1 个主视觉，但它不是孤立卡片。
2. 有 2–4 个分布在不同位置的辅助元素。
3. 有 1 条可解释的空间关系：路径、曲线、阶梯、轨道、环形、堆叠或左右对比。
4. 有 1 次状态变化：负载上升、节点接通、权限解锁、卡片替换、进度完成或颜色转折。
5. 可见元素占用至少 3 个空间锚点；静态关键帧能一眼看出这是“一个舞台”，而不是“人物旁边贴了说明”。

制作技术讲解、抽象概念、流程或诊断类舞台时，读取 [整屏舞台模式库](references/full-frame-stage-patterns.md)。其中包含空间锚点、人物避让、仪表盘、曲线路径、资料阶梯、工具网络和诊断轨道的可复用范式。

## 工作流

### 1. 盘点素材

先读取项目规则和当前目录。优先识别下面这种项目输入；文件名不必完全一致，但要能判断用途：

```text
本期视频/
├── 口播.mp4                 # 或 .mov；人物口播与原声
├── 口播稿.md                # 可选但推荐，用于校正专有名词和语义
└── 素材/
    ├── 01-deepseek-api.mp4
    ├── 02-对接-api.png
    ├── 03-增加-skill.mov
    └── 04-导入-skill.png
```

用 `ffprobe` 检查每段视频和素材的：

- 时长、分辨率、帧率、音频流
- 人物是否完整出镜、是否需要横向重构
- 口播是否包含明显停顿、重录、口误或需要保留的呼吸
- 是否有中间真实录屏，以及它需要接在哪一句口播之后
- 图片的尺寸、方向、透明通道和可读区域
- 素材是否需要作为全屏段、PIP、卡片、背景纹理或局部裁切使用

为每个素材建立简短索引：`assetId`、类型、语义、建议用法、可用时长、是否有音频、是否包含需要保护的文字 / UI。真实产品界面和用户提供的操作素材优先级高于自动生成的仿真界面。

不要移动、覆盖或删除用户原始素材。输出视频使用独立版本名，例如 `intro-v1.mp4`、`intro-v2.mp4`。

### 2. 建立口播时间轴

按以下顺序处理：

1. 从视频提取音频并生成字幕 / JSON 时间轴。
2. 用用户提供的逐字稿校正识别文字，尤其检查产品名、工具名和英文专有名词。
3. 标记句子开始、句子结束、停顿和关键词落点。
4. 将每句口播绑定到一个主要视觉动作，并标出该句的舞台类型、素材引用和节奏变化。
5. 对“说到结果 / 数字 / 工具名 / 操作动作”的关键词记录 payoff 时间，让对应元素在关键词落下时完成落位。

字幕识别结果可以只作为时间参考，不能直接当成最终文案。不能凭感觉猜时间，也不能用“每 3 秒一个画面”的方式代替口播时间轴。

### 3. 先写句子级分镜

制作前先列出下表；用户只要求分镜时，在这里停止，不渲染视频。

| 时间 | 口播含义 | 风格 | 舞台类型 | 人物 / 素材布局 | 主视觉 | 辅助元素与转场 | 真实素材 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 0:00-0:00 | 当前句子 | `clinical-tech` | 数据 / 流程 / 对比 / 实操 / 素材 / B-roll | 完整 / 左移 / 回中 / 全屏 | 一个主要图形或状态 | 2–4 个有语义元素，写明进入 / 强调 / 退出 | 文件名或无 |

分镜规则：

- 关键字出现前约 2-4 帧进入，发音落下时完成落位。
- 长句使用慢速展示或轻微推进；允许元素累积，但每次新增信息都必须服务口播，不连续堆叠同质卡片。
- 信息动画尽量放在人物空出的区域，避免覆盖脸、嘴和字幕。
- 录屏接入点要保留与录屏第一帧相同的窗口边缘、标题栏或位置关系。
- 抽象观点、反常识判断、情绪转折和需要换气的段落，可以标记为全屏 B-roll 候选；一条 B-roll 通常服务一个清晰隐喻，不要给每句话都配生成画面。
- 结尾保持少量信息层，方便下一段拼接或直接发布。

#### 舞台级分镜契约

每个舞台至少写清楚以下字段；没有字段就不要直接进入全片渲染：

```json
{
  "sceneId": "api-connection",
  "start": 18.42,
  "duration": 5.80,
  "styleProfile": "clinical-tech",
  "layoutProfile": "operation-upper",
  "engine": "hybrid",
  "spatialAnchors": ["top-left", "center-right", "lower-left"],
  "topology": "curved-flow",
  "stateChange": "connection-complete",
  "layers": [
    {"id": "section-tag", "role": "context", "appearAt": 0.00, "exitAt": 5.80},
    {"id": "main-flow", "role": "primary", "appearAt": 0.12, "emphasizeAt": 1.20, "exitAt": 5.50},
    {"id": "api-screen", "role": "asset", "assetId": "02-api", "appearAt": 0.56, "exitAt": 5.80},
    {"id": "result-chip", "role": "support", "appearAt": 2.10, "exitAt": 5.50}
  ]
}
```

- `role` 只允许 `context`、`primary`、`support`、`asset`、`ambient`、`subtitle`。
- `primary` 每个舞台只能有 1 个；`support` 通常 2–4 个；`ambient` 最多 1 个。
- `spatialAnchors` 至少 3 个；普通解释舞台至少跨越 2 列和 2 行。实操保护段可例外，但必须说明保护区。
- `topology` 写明元素关系，例如 `curved-flow`、`stack`、`ladder`、`orbit`、`split-compare`、`diagnostic-rail`。
- `stateChange` 写明舞台内唯一的主要变化；没有状态变化就不要进入实现。
- `styleProfile` 必须来自 [style-profiles.md](references/style-profiles.md) 已归档列表；当前只有 `clinical-tech`。项目实验风格在完成沉淀前不得伪装成正式 Profile。
- 每个 `layer` 都要有 `appearAt`，需要退出或替换时写 `exitAt`；强调、数值落点或状态变化写 `emphasizeAt` / `stateAt`。
- 进入顺序默认是：章节上下文 → 主视觉 → 真实素材 / 关键状态 → 辅助说明 → 收束。不要让所有元素同帧首现。

舞台类型和默认布局：

- `talking-head-editorial`：人物为主体，左 / 右空区承载标题、数据和标签。
- `full-frame-explainer`：人物为底色，3–5 个锚点分布全屏，用曲线、阶梯、堆叠或仪表盘形成完整解释舞台。
- `network-stage`：中心节点 + 四周能力 / 工具节点 + 动态连线，适合 Agent、工具、权限和系统组成。
- `diagnostic-stage`：多条诊断轨道分布在上下 / 左右区域，按口播逐项点亮，适合排查、清单和决策树。
- `operation-upper`：信息层只在上半区，原片操作模块保持完整。
- `comparison`：两列状态、中央连接线或箭头，差异点逐项高亮。
- `data-story`：大数字、图表 / 环形进度、标签和结论卡协同变化。
- `asset-pip`：真实图片 / 视频进入手机框、浏览器框或浮动卡片，旁边保留解释标签。
- `full-screen-asset`：素材占满画面，口播不断，退出时回到人物或下一舞台。

### 4. 选择实现方式

- **Remotion 与 HyperFrames 不是二选一**：按镜头和组件分工，同一条成片可以同时使用两个引擎。先确定一个主时间轴和最终合成者，再把另一个引擎产出的透明叠加层、全屏段或仿真 UI 作为媒体资产接入。
- **Remotion**：适合承载整片时间轴、口播视频、音频、字幕、PIP、录屏、SFX/BGM，以及横屏和竖屏的可复用 React 组件。已有 Remotion 项目时，复用已有 `Composition`、主题、`BaseVideo`、场景组件和渲染命令。
- **HyperFrames**：适合快速制作 HTML/CSS/GSAP 动态标题卡、数据图、仿真 UI 和信息层，也可以独立渲染全屏段。不要把它限制成“只做预览”。
- **混合桥接**：HyperFrames 输出透明 WebM/MOV 或无声普通 MP4，Remotion 用 `Video` / `OffthreadVideo` 放入对应 `Sequence`；Remotion 输出的透明动画或全屏段也可以反向作为 HyperFrames 的媒体素材。桥接时统一 `start`、`duration`、`fps`、`width`、`height` 和色彩空间。
- **唯一时间轴**：无论哪个引擎做镜头，`work/制作规格.md` 或 `timeline.json` 都是唯一 cue 来源；禁止两个引擎各自维护一套开始时间。最终合成者只保留一份人声、字幕和 BGM，叠加层默认无声。
- **舞台先于引擎**：先确定 `sceneId`、`layoutProfile` 和 `layers[]`，再为每个层选择 Remotion、HyperFrames 或 hybrid；不能因为换引擎而改变画面语义。
- **抽象观点需要独立的全屏视觉隐喻**：调用 `gbro-collage-broll`，生成半调纸拼贴组装式 B-roll，作为全屏切入或节奏穿插素材；它默认输出 9:16、约 5 秒、无声 MP4，不负责透明图层和逐层可编辑时间线。
- 不要为了换工具而重建已经可用的项目。当前项目已有 Remotion 时，继续在当前 Remotion 项目中实现。
- 不要把用户要求的“纯动画”误解成“生成一张图片再放进去”。动画中的文章、窗口和标签都应由代码绘制。

#### 全屏穿插型拼贴 B-roll

当分镜把某句口播标记为“全屏 B-roll”时，按以下方式接入：

1. 先从逐字稿和时间轴中确定候选句、插入点、预计时长和它要表达的单一隐喻。
2. 调用 `gbro-collage-broll` 前运行其环境自检；该 Skill 的三道闸门必须保留：先确认视觉隐喻，再确认彩色静帧，最后才生成视频。
3. Gate 3 生成的 B-roll 默认使用无声版本，原口播音轨保持连续；优先在句子边界、停顿或关键词落点切入，避免为了塞画面而打断语义。
4. 将通过 QA 的 B-roll 作为独立画面轨插回成片，可做全屏 cutaway；如果需要局部叠加、透明背景、人物旁边漂浮或精确控制遮挡，改回 Remotion / HyperFrames 实现。
5. 回插后重新检查：原声是否连续、切入是否落在正确句子、B-roll 是否出现黑帧或多余音轨、9:16 裁切是否安全，以及全屏画面是否抢走了口播核心信息。

不要把普通录屏、产品操作演示或需要精确文字的画面交给拼贴 B-roll；这些内容继续使用用户提供的真实录屏或代码绘制的信息层。

### 5. 实现动画

制作 Winter 的 WorkBuddy / AI 工具类竖屏口播时，优先复用 [WorkBuddy 竖屏信息层规范](references/workbuddy-vertical-overlay.md)。该规范记录了已验证的上下区域分配、临床深色玻璃风格、字号层级、步骤卡、统计环、运动节奏和高质量渲染参数。

风格选择和 tokens 读取 [style-profiles.md](references/style-profiles.md)；技术讲解、抽象概念、流程或诊断类整屏舞台读取 [full-frame-stage-patterns.md](references/full-frame-stage-patterns.md)。

制作横屏或混合引擎视频时，读取同一规范中的 [Remotion / HyperFrames 混合桥接](references/workbuddy-vertical-overlay.md#remotion--hyperframes-混合桥接)；横屏只改变布局 Profile，不改变 cue、元素运动和质量检查规则。

#### 舞台与图层实现

每个 `scene` 作为一个独立视觉舞台实现，不要只生成一张背景卡。推荐结构：

```text
Stage
├── ContextLayer       # 章节标签 / 小标题 / 进度
├── PrimaryLayer       # 当前口播唯一主视觉
├── SupportLayers[]    # 数据、标签、箭头、对比、解释
├── TopologyLayer      # 曲线路径 / 阶梯 / 轨道 / 节点连线
├── AssetLayer         # 用户提供的图片 / 视频 / 录屏
├── AmbientLayer       # 低注意力的 glow、网格、粒子或色彩呼吸
└── SubtitleLayer      # 最后合成的字幕
```

每个图层使用同一套生命周期：

```text
appear → settle → hold → emphasize/state-change → exit/replace
```

编排规则：

- 主视觉先落位，辅助层错峰进入；可以并行运动，但不能让两个独立信息同时争夺第一注意力。
- 先画 `TopologyLayer`，再放主视觉与辅助元素。路径或结构先决定视觉流向，不能先把卡片堆在角落再补装饰线。
- 普通舞台先划分 `top-left / top-right / center-left / center-right / lower-left / lower-right` 锚点，选择其中 3–5 个；人物脸和嘴是避让区，不是默认中心锚点。
- 细曲线、虚线和低对比路径可以经过人物背景；大型文字卡和主节点不得长时间覆盖眼睛、嘴和字幕。
- 一个舞台至少安排一次有意义的状态变化：数值增长、节点高亮、箭头完成、卡片替换、素材推进或颜色转折。
- 画面稳定阅读时保留至少约 1 秒；复杂流程和真实界面按 1× 播放速度可读性决定停留时间。
- 舞台切换可以采用淡出 / 滑入 / 交叉替换 / 全屏切入 / 素材卡带入，但不能只依赖统一的卡片从下方弹出。
- 退出时优先保留主视觉的结论，再让辅助层退出；下一舞台可复用同一位置，形成连续叙事。
- 任何装饰元素都必须能解释其作用：章节定位、层级分组、路径指向、状态反馈或节奏衔接；否则删除。

推荐把信息层拆成可复用组件：

- `BaseVideo`：口播视频、缩放和按场景横向偏移
- `Stage`：按 `layoutProfile` 管理一组图层、注意力顺序和舞台转场
- `LayerTimeline`：统一处理 `appearAt`、`holdUntil`、`emphasizeAt`、`stateAt`、`exitAt`
- `MotionLayer`：单个图层的进入、强调、替换、退出和统一字体
- `ArticleMock`：标题、正文、引用、代码块和局部高亮
- `ManualEditorMock`：字体、间距、代码块的逐项调整和计时器
- `PublishCompare`：编辑器预览与发布后预览
- `EditorMock`：左侧输入文章、右侧直接预览
- `ProgressBar`：开头、中间、结尾的章节进度
- `AssetFrame`：图片 / 视频 / 录屏的手机框、浏览器框、PIP、裁切和安全区
- `DataStory`：数字、图表、环形进度、节点和结果状态的联动动画
- `StageTopology`：SVG 曲线、轨道、阶梯、节点连接和路径生长
- `MetricGauge`：环形负载、能力边界、信号衰减和权限状态
- `DataStack`：资料堆叠、层级累积、拥挤与重要信息淹没
- `NodeNetwork`：中心节点、四周工具节点、连线接通和权限切换

实现时遵循：

- 所有中文使用正常 DOM / JSX 文本，不把关键文字做成不可校验的位图。
- 文字组件只使用平移、缩放和透明度动画；需要旋转时，旋转不包含任何文字。
- 信息对象进入后保留足够时间让观众读清楚，再进行下一次切换；多元素只是增加层次，不降低阅读时间。
- 用纯 CSS 线框、面板、纸张、滑杆和高亮框表达结构，不制造没有语义的装饰。
- 用户提供的素材必须进入 `AssetLayer` 并写明 `assetId`；禁止用近似的假 UI 替代能够直接使用的真实截图 / 录屏。
- 右侧动画较宽时，让人物向左移动；调整的是口播视频包装层，不修改原始视频。
- 回插全屏 B-roll 时只替换画面轨，不替换口播音轨；B-roll 的末尾要自然回到人物或下一段真实素材，避免连续多个生成画面造成视觉疲劳。
- 原片为 3:4、交付为 9:16 时，默认完整保留原画面并用同源虚化向上下延展，不裁掉人物手势或原片内嵌操作模块。需要 4K 交付时使用 `1080×1920` 逻辑坐标设计，再等比放大到 `2160×3840` 渲染。
- 实操段使用上半屏专用组件；在 1080×1920 逻辑坐标中，把大型信息卡限制在 `y=180–800`，以 `y=850` 作为操作区边界提示，下方只保留原片内容。
- 同一条视频混用两个引擎时，先把每个场景标注为 `engine: remotion`、`engine: hyperframes` 或 `engine: hybrid`，再决定输出透明叠加、全屏视频还是最终合成。引擎选择不能改变场景语义和质量验收标准。

### 维护与扩展

- GitHub 私有仓库 `winter-video-create` 作为 Skill 真源；视频、渲染结果、密钥和项目素材不进入仓库。
- 修改核心契约时，先更新 `SKILL.md`，再同步更新 `style-profiles.md`、`full-frame-stage-patterns.md` 或 WorkBuddy 参考规范。
- 新增风格必须来自已经完成并复用过的真实项目；达到沉淀标准后才修改风格 Profile 和组件适配。新增舞台只修改模式库和 `layoutProfile`，不要复制主流程。
- 每次变更至少运行 `quick_validate.py`，并用一个普通舞台、一个数据舞台、一个实操舞台做快速验证。
- 发布前检查 GitHub 与本地运行副本是否一致，提交信息说明是“规范变化”“新风格”还是“渲染实现变化”。

### 6. 渲染和验收

完成后必须执行：

1. Remotion 项目运行 lint 和 TypeScript；HyperFrames 项目运行一次 `check`（已经包含 lint），不要紧接着重复运行独立 lint。
2. 渲染独立版本，不覆盖之前能用的版本。
3. 用 `ffprobe` 检查输出的时长、分辨率、帧率和音频流。
4. 用 `ffmpeg` 的 `blackdetect` 检查是否出现意外黑帧或空白帧。
5. 快速预览只抽取首、中、尾 3 帧；正式成片抽取开头、结尾、章节代表帧和所有关键实操段代表帧。
6. 逐张检查：中文字是否正向清晰、人物脸部是否被遮挡、信息是否越界、动画是否与口播对应、录屏接口是否自然。
7. 对每个章节至少抽查一个完整舞台的连续帧：确认不是单卡片静止画面，至少能看到主视觉与辅助元素的时序关系，以及一次状态变化或有意义的转场。
8. 检查图层数量是否服务语义：普通解释舞台通常为 3–6 个可见元素；片尾、短句和操作保护段可以少于 3 个；超过 7 个时检查是否信息过载。
9. 做“角落聚集检查”：如果 70% 以上的信息面积集中在同一象限，或连续两个舞台复用同一角落卡片结构，判定为舞台失败并重做。
10. 抽取 6–13 个章节代表帧组成 contact sheet；缩小观看时仍应看出不同舞台拥有不同的空间骨架，而不是同一模板换文案。
11. 用 `rg` 搜索源码，确认纯动画版本没有误引用截图、失效素材或旧组件。

代码检查通过不代表视频合格。真实画面检查是必须步骤；发现问题就修改、重新渲染和再次抽检，最多连续自检 3 轮，仍有问题时明确报告残留风险。

使用全屏 B-roll 时额外检查：B-roll 首尾帧是否干净、是否为无声版本、与口播句子是否对应、切入和切出是否发生在自然停顿处，以及它是否仍然服务于口播而不是变成独立的装饰片段。

#### HyperFrames 性能与高质量交付

- 优先设置 `HYPERFRAMES_BROWSER_PATH='/Applications/Google Chrome.app/Contents/MacOS/Google Chrome'` 使用系统 Chrome，避免损坏或重复下载无头浏览器缓存。
- 送入 HyperFrames 的长视频必须使用稳定关键帧：30fps 素材设置 `-g 30 -keyint_min 30 -sc_threshold 0`。出现 `sparse keyframes` 警告时立即停止渲染并重编码，避免并行 seek 冻帧。
- 用户要求“不压缩质量”时，默认以原素材横向分辨率为基准，不先降到 1080P；可使用 `--quality high --crf 10 --fps 30 --workers 4` 输出高质量 H.264。
- 4K + CRF 10 的慢速编码耗时明显高于帧捕获，进程仍高 CPU 且有心跳时继续等待，不误判为卡死。
- 最终黑帧检测可以先缩小检测分辨率，只用于判断整段是否意外全黑；无需用 4K 原尺寸执行像素扫描。
