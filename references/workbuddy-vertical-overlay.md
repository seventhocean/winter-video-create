## 使用范围

用于人物口播已经拍好、原片可能内嵌真实产品操作、需要通过 Remotion 或 HyperFrames 增加信息图的 AI 工具类竖屏视频。

本规范来自一次已完成的 111.3 秒 WorkBuddy 答疑视频：共 34 个句子级视觉节点，包含积分 / Token、Skill、上下文三个章节及四段原片内嵌实操。

## 舞台化多层编排

### 核心模型

每一次转场、每一个节奏段都视为一个独立舞台，而不是一张孤立卡片。一个普通舞台默认由以下层组成：

```text
Stage
├── ContextLayer       # 章节、眉题、进度
├── PrimaryLayer       # 当前口播的唯一主视觉
├── SupportLayers[]    # 数据、标签、箭头、对比、解释
├── AssetLayer         # 用户提供的图片、视频、录屏
├── AmbientLayer       # 低注意力的背景呼吸或光线
└── SubtitleLayer      # 最终字幕，不参与信息层抢位
```

普通解释舞台通常保持 3–6 个可见元素；短句、片尾和操作保护段可以更少。不要为了数量加入无语义粒子、噪点或随机卡片。

### 推荐布局 Profile

| Profile | 主画面 | 辅助区域 | 典型用途 |
| --- | --- | --- | --- |
| `talking-head-editorial` | 人物 + 左 / 右主标题 | 标签、数据、连线、章节条 | 观点、结论、开场 |
| `operation-upper` | 上半屏信息层 | 下半屏保留真实操作 | API、Skill、导入、配置 |
| `comparison` | 左右双状态 | 中央箭头 / 差异标签 | 前后对比、方案比较 |
| `data-story` | 大数字 / 图表 | 环形进度、结论卡、时间线 | 增长、价格、比例、趋势 |
| `asset-pip` | 人物 + 图片 / 视频卡 | 手机框、浏览器框、说明标签 | 用户提供的截图和素材 |
| `full-screen-asset` | 素材全屏 | 少量章节标识 | 真实录屏或全屏 B-roll |

### 图层生命周期

所有非字幕元素都按下面的生命周期设计：

```text
appear → settle → hold → emphasize/state-change → exit/replace
```

- 主视觉先落位，辅助元素错开进入；同一时刻只能有一个新信息获得第一注意力。
- 多个元素可以同时运动，但同时运动的元素必须属于同一语义动作，例如“数字增长 + 环形进度展开”。
- 每个舞台至少设计一次语义状态变化：数字增长、节点高亮、连线完成、卡片替换、素材推进或色彩转折。
- 复杂元素完成进入后至少稳定停留约 1 秒；口播较长时可以持续轻微运动，但不能用漂浮掩盖没有内容变化。
- 舞台退出时先留下结论，再移除辅助信息；下一舞台可以复用位置，但要有明确的状态替换或转场关系。

### 舞台级契约

分镜和实现都使用同一份 `layers[]` 数据，不让 Remotion 和 HyperFrames 分别维护时间：

```json
{
  "sceneId": "api-connection",
  "layoutProfile": "operation-upper",
  "layers": [
    {"id": "section-tag", "role": "context", "appearAt": 0.00, "exitAt": 5.80},
    {"id": "main-flow", "role": "primary", "appearAt": 0.12, "emphasizeAt": 1.20, "exitAt": 5.50},
    {"id": "api-screen", "role": "asset", "assetId": "02-api", "appearAt": 0.56, "exitAt": 5.80},
    {"id": "result-chip", "role": "support", "appearAt": 2.10, "exitAt": 5.50}
  ]
}
```

字段约束：

- 每个舞台只有 1 个 `primary`；通常有 2–4 个 `support`，最多 1 个 `ambient`。
- 每个图层必须有 `appearAt`；需要退出或被替换时必须有 `exitAt`。
- 关键词落点、数字完成、按钮点击或状态切换写入 `emphasizeAt` / `stateAt`。
- 真实图片、视频和录屏必须通过 `assetId` 接入；可做 PIP、手机框、浏览器框、全屏段或局部裁切，但不能被近似假 UI 替代。

## 画布与区域分配

### 逻辑坐标

- 设计坐标：`1080×1920`。
- 4K 交付：整体等比放大 2 倍至 `2160×3840`，不要先把 2160 宽原片降采样。
- 3:4 原片进入 9:16 时：原画面完整居中；逻辑坐标上下各保留约 240px 同源虚化延展。禁止为了铺满画布裁掉原片内的操作内容。

### 普通口播段

- 左右安全边距：56px。
- 信息卡常用顶部：180px。
- 常规卡最小高度：520px；章节、情境和片尾卡可扩展至约 720px。
- 信息卡可以遮挡额头、眼睛或脸部，但应避免长期覆盖嘴部；情绪收束和片尾可回到人物主画面。

### 原片内嵌实操段

- 固定优先级：`真实操作模块 > 信息动效 > 人脸`。
- 大型信息动效仅进入逻辑坐标 `y=180–800`。
- 在 `y≈850` 放置细线和“下方为真实操作区域”提示；不得让卡片越过该边界。
- `y≥900` 默认视为操作保护区：按钮、输入框、下拉选项、光标、弹窗和保存结果必须完整可见。
- 顶部可以增加章节条和 `STEP 01 / 02 / 03`，但不要在下方操作区叠加箭头、字幕底板或装饰粒子。

### 横屏布局 Profile

横屏使用 `1920×1080` 逻辑坐标；不改变舞台、图层和时间码契约，只改变区域分配：

- 左右安全边距建议 64–88px；顶部保留章节眉题和小型进度条。
- `talking-head-editorial` 默认把主标题 / 数据放在左侧，把人物和素材卡放在右侧；人物偏移只作用于包装层。
- `asset-pip` 可将手机、网页或图片素材放在右侧 28%–38% 区域，左侧保留主观点和一条结论标签。
- `comparison` 使用左右双列或上下双列；中央只保留一条清晰连接关系，不堆多条无意义箭头。
- 字幕安全区固定在底部约 120–170px；任何舞台层不得压住字幕底板。
- 全屏素材可以短暂占满画面，但切出前要保留人物或章节定位，避免观众失去叙事锚点。

## Remotion / HyperFrames 混合桥接

两者不是替代关系，而是同一条制作管线里的两个执行层。按镜头选择引擎，保持一份统一时间轴。

### 推荐分工

| 层 | Remotion | HyperFrames |
|---|---|---|
| 主时间轴 | 适合承载口播、字幕、PIP、录屏、SFX/BGM | 也可以承载独立信息图或全屏段 |
| 视觉模块 | React 场景、横屏 / 竖屏包装、复杂合成 | HTML/CSS/GSAP 数据图、步骤卡、仿真 UI、动态标题 |
| 输出形式 | 最终 MP4，或透明动画资产 | 透明 WebM/MOV、无声 MP4、全屏段 |
| 适用目的 | 长视频、固定栏目、正式合成 | 快速迭代、精细信息层、可独立复用的动效模块 |

### 同片混用的三种模式

1. **Remotion 主片 + HyperFrames 叠加层**：Remotion 负责口播视频、字幕和声音；HyperFrames 输出无声透明动画，在 Remotion 的对应 `Sequence` 中叠加。适合 API 价格卡、进度环、步骤卡和仿真 UI。
2. **Remotion 主片 + HyperFrames 全屏段**：HyperFrames 输出普通 MP4，Remotion 在句子边界切入作为全屏信息图或节奏段；原人声由 Remotion 持有，HyperFrames 文件不带音轨。
3. **HyperFrames 主片 + Remotion 资产**：当整片已经是 HTML/GSAP 时间轴时，把 Remotion 生成的透明动画或全屏段作为媒体素材接入；适合已有 HyperFrames 工程不值得重建的情况。

### 桥接契约

每个跨引擎资产必须记录：

```json
{
  "sceneId": "token-price",
  "engine": "hyperframes",
  "start": 21.16,
  "duration": 3.62,
  "fps": 30,
  "width": 2160,
  "height": 3840,
  "alpha": true,
  "audio": false,
  "safeZone": "portrait-upper"
}
```

- `work/制作规格.md` 或 `timeline.json` 是唯一时间码来源。
- 所有引擎使用同一 fps、分辨率、画幅和色彩空间；不要把 30fps 的 HyperFrames 资产塞进 60fps 主片后再靠播放器补帧。
- 透明叠加层默认无声；人声、字幕、SFX 和 BGM只由最终合成者保留一份。
- 不需要透明时，优先输出普通 MP4 全屏段，避免为了 alpha 增加编码复杂度。
- 叠加层要在桥接前确认 premultiplied alpha、边缘黑边、色彩范围和首尾帧；发现黑边或透明失真时，改用全屏普通视频或 PNG 序列。
- 最终合成者可以是 Remotion，也可以是 HyperFrames；但每条成片只能有一个最终音画封装出口。

### 引擎选择示例

- 横屏口播：Remotion 放主讲人、字幕和章节条；HyperFrames 生成右侧 API 价格图和浏览器仿真窗口。
- 竖屏实操：Remotion 保留整条口播和音频；HyperFrames 只生成 `y=180–800` 的上半屏信息卡，下半屏原操作模块保持原样。
- 复杂全屏解释段：HyperFrames 先独立制作并确认节奏，再作为无声 MP4 插入 Remotion 的句子边界。

## 视觉语言

### 基础风格

- 风格：临床深色科技编辑感，避免游戏 HUD 和无意义霓虹。
- 背景卡：`rgba(4,17,31,.88)`。
- 主文字：`#f5f9ff`；次文字：`#afc0d2`。
- 主强调：亮蓝 `#32aeff`；完成状态：绿色 `#42df86`。
- 警告：黄色 `#ffc34d`；错误 / 低效：红色 `#ff6577`。
- 边框：2px、低透明亮蓝；圆角约 28px；背景模糊约 24px；阴影向下扩散，不使用厚重外发光。

### 字体层级（1080×1920 逻辑坐标）

- 顶部章节标签：20–22px、粗体、字距约 3px。
- 说明眉题：18–20px、粗体、字距约 3px。
- 主标题：66–76px；片尾或单一观点可到 92px。
- 数据大字：150–190px，使用等宽数字特性。
- 正文说明：27–30px，行高约 1.4。
- 芯片 / 按钮：20–22px、粗体。

## 元素组件

### 统计命中

- 使用超大数字 + 单位 + SVG 环形进度。
- 数字负责第一眼信息，环形进度负责运动，不再叠加无来源数据。
- 适合 `99%`、`1%`、`100积分`、`<¥2/天` 等口播明确给出的数字。

### 步骤卡

- 左侧窄栏展示超大 `STEP 01 / 02 / 03`。
- 右侧依次放标题、两到三个步骤芯片、水平进度线。
- 实操时只说明当前动作，不复刻下方页面全部文字。

### 流程与对比

- 用三个节点表达“任务 → 卡点 → 专用 Skill”。
- 用双列状态表达“信息不足 / 信息完整”“高成本 / 低成本”。
- 一张卡只承载一个判断，不把多个观点塞入同一面板。

### 情境与沟通

- 使用两条对话气泡模拟人和 AI 的沟通。
- Prompt 场景用浏览器 / 编辑器线框与逐行高亮表达信息补充。
- “多用 ↔ 多沟通”使用闭环结构，但只旋转图形，不旋转文字。

### 状态与收尾

- 完成状态使用绿色对勾、进度条和三章节圆点。
- 片尾减少元素，只保留 `WINTER`、下一话题和“下期见”。

## 运动节奏

- 一句口播绑定一个主要视觉动作；一个舞台允许 2–4 个辅助层围绕主动作分批出现。
- 34 个视觉节点覆盖 111.3 秒，平均约 3.3 秒一个节点；实际按语义长短分配，不机械等分。
- 舞台进入顺序默认是“章节上下文 → 主视觉 → 真实素材 / 关键状态 → 辅助说明 → 收束”，不要所有元素同帧首现。
- 卡片进入：约 0.45–0.55 秒，常用 `expo.out`。
- 标题进入：约 0.4–0.5 秒，水平位移 35–45px，常用 `power4.out`。
- 芯片 / 图标：约 0.3–0.4 秒，间隔 0.06–0.1 秒。
- 卡片退出：约 0.25–0.3 秒，轻微上移并淡出。
- 读数、进度环和进度线在 0.7–1.2 秒内完成，之后稳定停留供阅读。
- 舞台转场不固定为一种动作：可用交叉替换、横向滑入、素材卡带入、状态替换、全屏切入或章节色彩转折；同一段不要连续重复同一种弹出方式。
- 长句可以让背景 glow 缓慢位移维持轻微生命感；不要增加与语义无关的持续漂浮。

## 快速预览与正式成片

### 快速预览

- 截取 10–15 秒，必须包含一个普通信息图和一个真实实操画面。
- 使用 1080×1920、standard 质量即可。
- 只检查首、中、尾 3 帧以及操作区遮挡关系。
- 至少选一个舞台检查连续帧，确认主视觉、辅助层和状态变化按顺序发生，而不是静态单卡片。
- 风格未确认前不要转完整 4K 中间素材，不做整片黑帧扫描。

### 正式成片

1. 保留原始横向分辨率，转成 9:16 高质量中间素材。
2. 30fps 素材设置 1 秒关键帧间隔：

```bash
ffmpeg -i input.mov \
  -c:v libx264 -crf 10 -g 30 -keyint_min 30 -sc_threshold 0 \
  -c:a aac -b:a 320k -movflags +faststart output-hq.mp4
```

3. 若需同源虚化扩展，先生成 9:16 模糊底图，再把原 3:4 画面完整居中覆盖。
4. HyperFrames 使用系统 Chrome：

```bash
HYPERFRAMES_BROWSER_PATH='/Applications/Google Chrome.app/Contents/MacOS/Google Chrome' \
PRODUCER_BROWSER_GPU_MODE=hardware \
npx hyperframes render --quality high --crf 10 --fps 30 --workers 4 \
  --output renders/final-4k.mp4
```

5. `sparse keyframes` 是阻断性警告：立即停止正式渲染并重编码输入素材。
6. 交付前确认分辨率、时长、30fps、音轨；抽查开头、两个实操区、章节代表帧和片尾。

## 已验证的质量基准

- 2160×3840、30fps、H.264 CRF 10。
- 111.3 秒成片约 689MB，视频码率约 51.7Mbps。
- 四个并行 worker 的完整本地渲染约 15 分钟，其中慢速高质量编码是主要耗时。
- 该数字只用于估算机器耗时和存储，不应成为所有项目的固定文件大小目标。
